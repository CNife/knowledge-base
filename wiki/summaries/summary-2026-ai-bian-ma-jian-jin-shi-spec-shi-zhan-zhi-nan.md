## 主要观点

提出"渐进式 Spec Coding"框架，核心主张：不同复杂度的需求应暴露不同深度的流程——简单需求不承担复杂流程的成本，流程是可选增强而非强制前提。这本质上是在压缩偶然复杂度：只有本质复杂度够高时，才引入对应重量的流程。

## 关键细节

- **模型梯队差异**：T0 模型一次生成全链路且主动处理边界情况；T1 多提示一两轮可达到接近效果；T1.5 基本可用但容易漏边界；T2 能写骨架但需要较多人工调整。差距不在"能不能写"，而在一次做对的概率——T0 三轮搞定的事，T2 可能 15 轮还不一定对。
- **Agent 本质**：Agent = while 循环 + Tool Use + 工具执行器。"智能"来自模型，"能力"来自工具，"自主性"来自循环。工具的边界就是 Agent 的能力边界。
- **Spec Coding 三条铁律**：No Spec, No Code（没有文档，不准写代码）；Spec is Truth（文档和代码冲突时，错的一定是代码）；Reverse Sync（发现 Bug，先修文档，再修代码）。
- **渐进式复杂度**：70% 的需求是 ≤5 人日的小需求。改个字段、修个 bug，也要先写 spec 再拆 tasks？这就是偶然复杂度在吃掉效率。不同复杂度的需求，暴露不同深度的流程。
- **编排层 + 执行层两层架构**：编排层 AI（强模型）负责理解需求、生成 Spec、审查结果；执行层 AI（编码工具）负责读写代码、执行命令、运行测试。分层后各取所长，成本和质量都更优。
- **透明度底线**：模型型号+版本可见、完整 context 可查、原始输出不被篡改、token 用量透明。在不透明的工具上花再多时间优化 prompt 和框架，效果都无法归因、无法复现。
- **知识飞轮**：需求实践 → 踩坑 → 沉淀 knowledge / 更新 prompt / 修改模板 → AI 更准 → 更好的实践。不仅是领域知识，prompt 和模板自身也在飞轮中。
- **自由度曲线**：调研阶段自由度中等（让 AI 自由探索，但必须给证据）；方案设计阶段自由度高（唯一鼓励 AI 充分想象的阶段）；规划阶段自由度低（精确到文件路径和函数签名）；执行阶段自由度零（严格按计划施工）；验收阶段自由度中等（自由检查，但结论要有依据）。

## 方法 / 进路

### Spec Coding 框架设计

框架核心设计：渐进式复杂度——不同复杂度的需求，暴露不同深度的流程。

关键原则：
- 简单需求不承担复杂流程的成本——改个字段不需要先写 spec 再拆 tasks
- 流程是可选增强，而非强制前提——Rules 始终生效，Spec 按复杂度加载
- 这本质上是在压缩偶然复杂度：只有本质复杂度够高时，才引入对应重量的流程

### 工作流：Propose → Apply → Review → Archive

**Propose（提案）**：人主导，AI 辅助
- Research：分析代码现状，锁定事实（每个结论有代码出处）
- 逐个提问：一次只问一个问题或一组紧密相关的问题，优先给 2-3 个选项 + 推荐
- 分段生成文档，每段确认：不一口气生成完整 spec
- 生成完整 spec.md + tasks.md + log.md
- HARD-GATE 确认：完整 spec + tasks 生成后，必须等用户显式确认

**Apply（执行）**：AI 主导，人审查
- 默认逐步执行：完成一个 task → 报告 → 等用户确认
- 零偏差原则：Plan 是合同，AI 是打印机
- Verification 铁律：每个 task 完成后必须展示可验证的证据

**Review（审查）**：两阶段 Sub Agent 审查
- Spec Compliance（spec-reviewer）：逐条比对 spec 功能点与实际代码
- Code Quality（code-quality-reviewer）：基于 rules/ 检查编码规范、安全红线

**Archive（归档）**：知识沉淀
- 逐条展示 log.md 中的知识发现和踩坑记录，询问用户是否沉淀到 knowledge/

## 结果 / 证据

文章提供了完整的实战案例：跨模块业务迁移项目（将系统 A 的过滤处理逻辑平行迁移到系统 B），展示了完整的 /propose → /apply → /review 流程。

关键证据：
- AI 在 Propose 阶段先 Research 代码现状，然后逐个提问收敛不确定性
- 待澄清项全部解决前不进入编码
- Apply 阶段按 Tasks 逐个执行，每完成一批文件展示完成状态和核心实现结构
- Reverse Sync 在执行过程中保持文档和代码的一致性

## 局限性

文章承认的局限性：
- **心流中断**：编码需要高度专注。引入 AI 之后，连贯性被打破了——写 Spec、等生成、审查输出、纠正偏差、再继续下一段。模型越慢，等待越长，心流杀伤越大。
- **上下文的隐性成本**：Context 很贵，但贵的不只是 Token 费用。更大的隐性成本是上下文管理本身：你需要有意识地决定给 AI 看什么、不看什么。
- **现在不是终态**：编程可能是 AI 产生飞轮效应最直接的领域。今天某些模型的速度和质量让你觉得勉强够用，半年后可能完全不同。

观察到的局限性：
- 文章主要针对 Java 后端项目，对其他语言和领域的适用性需要验证 [未验证]
- 框架的复杂度分级标准（简单/中等/复杂）缺乏明确的量化定义 [未验证]
- 知识飞轮的效果需要长期实践验证 [未验证]

## 值得保留的引用

> 模型是地基，方法论是上层建筑。地基不行，上面盖得再好也白搭。

> Agent = while 循环 + Tool Use + 工具执行器。"智能"来自模型，"能力"来自工具，"自主性"来自循环。

> 软件复杂度 = 本质复杂度（业务逻辑本身，不可消除）+ 偶然复杂度（工具/流程引入的额外负担，可以且应该被消除）

> 代码是廉价的消耗品，文档（Spec）才是昂贵的核心资产。

> 一个没有 knowledge/ 的 Spec 框架，就像让一个刚入职的应届生对着编码规范写代码——规范他都能遵守，但业务逻辑全靠猜。

> 知识底座才是真正的护城河。团队之间的差距不在于用什么工具，而在于积累了多少高质量的、结构化的领域知识。
