# 渐进式编码

## 定义

不同复杂度的需求，暴露不同深度的流程——简单需求不承担复杂流程的成本，流程是可选增强而非强制前提。核心思想：只有本质复杂度够高时，才引入对应重量的流程。

## 为什么重要

其他 Spec Coding 方案都假设所有需求都值得走完整流程，但现实中并非如此——70% 的需求是 ≤5 人日的小需求。改个字段、修个 bug，也要先写 spec 再拆 tasks？这就是偶然复杂度在吃掉效率。

渐进式编码解决了这个问题：简单需求不承担复杂流程的成本，Rules 始终生效，Spec 按复杂度加载。这本质上是在压缩偶然复杂度。

## 工作原理

### 复杂度分级

| 复杂度 | 流程深度 | 适用场景 |
|--------|----------|----------|
| 🟢 简单 | Rules + 直接编码 | 改字段、修 bug、小调整（≤1 人日） |
| 🟡 中等 | Rules + Spec + Tasks | 新功能、重构、跨模块修改（2-5 人日） |
| 🔴 复杂 | Rules + Spec + Tasks + Review + Archive | 大型迁移、架构调整、核心逻辑变更（≥5 人日） |

### 关键原则

1. **简单需求不承担复杂流程的成本**——改个字段不需要先写 spec 再拆 tasks
2. **流程是可选增强，而非强制前提**——Rules 始终生效，Spec 按复杂度加载
3. **压缩偶然复杂度**——只有本质复杂度够高时，才引入对应重量的流程

### 目录结构

```bash
code_copilot/
├── rules/                      # Project Rules（始终生效）
├── knowledge/                  # 领域知识（按需加载）
├── agents/                     # Agent 配置与提示词
├── changes/                    # 变更管理（按复杂度加载）
└── archives/                   # 已完成变更的归档
```

## 关键属性 / 权衡

- **效率提升**：简单需求直接编码，不承担 Spec 流程成本
- **质量保障**：Rules 始终生效，确保编码规范和安全红线
- **灵活性**：流程是可选增强，而非强制前提
- **可扩展**：复杂度分级可根据团队实际情况调整
- **风险**：复杂度分级标准需要明确量化定义 [未验证]
