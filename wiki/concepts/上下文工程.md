# 上下文工程

## 定义

上下文工程是控制模型能看到什么信息的系统性方法，包括分层管理、压缩策略、Prompt Caching 优化和 Skills 按需加载。核心目标是保持信息密度，避免无关内容稀释关键信号（Context Rot）。

## 为什么重要

Transformer 注意力复杂度是 O(n²)，上下文越长，关键信号越容易被噪声稀释。Agent 最常见的失效模式是无关内容占到上下文的大头后，决策质量明显下滑。问题通常不是窗口不够长，而是信息密度不对。[[agent-loop]] 的稳定性直接依赖于上下文工程的质量。

## 工作原理

上下文分层：
1. **常驻层**：身份定义、项目约定、绝对禁止项，每次会话必须成立，短、硬、可执行
2. **按需加载层**：Skills 和领域知识，描述符常驻，完整内容触发时注入
3. **运行时注入层**：当前时间、渠道 ID、用户偏好等动态信息
4. **记忆层**：跨会话经验写入 MEMORY.md，不直接进系统提示
5. **系统层**：Hooks 或代码规则处理确定性逻辑，完全不进上下文

压缩策略：

| 策略       | 成本   | 丢什么              | 适用场景               |
|-----------|-------|-------------------|----------------------|
| 滑动窗口   | 极低   | 早期上下文          | 简短对话              |
| LLM 摘要   | 中     | 细节，保留决策       | 长任务、含关键决策     |
| 工具结果替换| 极低   | 工具原始输出         | 工具调用密集型        |

Prompt Caching：
- 底层原理：前缀匹配缓存，KV cache 从缓存读取而非重新计算
- 命中前提：精确前缀匹配，任何一个 token 不同都会破坏匹配
- 缓存友好内容：系统提示、工具定义、长文档（多轮请求里基本不变）
- 动态信息位置：放在后面，不影响前缀稳定性
- 成本对比：稳定的大系统提示比频繁变动的小提示成本更低（写入成本只付一次，后续 90% 折扣）

## 关键属性 / 权衡

- **MCP 工具定义开销**：一个典型 MCP Server 约 4,000-6,000 tokens，5 个 Server 合计 25,000 tokens（12.5%）
- **Skills 描述符成本**：每个启用的 Skill 描述符常驻上下文，优化前约 45 tokens，优化后约 9 tokens
- **压缩保留优先级**：架构决策（不得摘要） > 已修改文件和关键变更 > 验证状态 > 未解决 TODO > 工具输出（可删）
- **标识符不可改动**：UUID、hash、IP、端口、URL、文件名必须原样保留
- **Cursor MCP 优化结果**：工具描述同步到文件夹，token 消耗减少 46.9%
